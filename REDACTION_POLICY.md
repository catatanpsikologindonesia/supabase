# Redaction Policy

This policy defines what must be removed or masked before sharing repository artifacts outside the trusted team boundary.

## Redaction Scope

Apply redaction checks to:
- `snapshot/config/*`
- `snapshot/database/*`
- `snapshot/storage/*`
- `snapshot/verification/*`
- Any logs generated by scripts or CI

## Must Redact

Always remove or mask:
- API keys, service keys, access tokens, refresh tokens
- Database credentials and connection strings with passwords
- JWT secrets, signing keys, private keys, certificates with private material
- PII and health-related personal data
- Internal hostnames, private IP ranges, and infrastructure identifiers when not required

## Masking Standard

- Use deterministic masking where possible (`<REDACTED_API_KEY>`, `<REDACTED_PASSWORD>`).
- Preserve structure and key names so troubleshooting context remains useful.
- Do not partially expose secrets; mask full values.

## Publication Checklist

Before pushing to public or shared remotes:
1. Review `git diff --cached` and all added files manually.
2. Scan for common secret patterns (tokens, passwords, bearer headers, private key blocks).
3. Remove transient error/log files that may include credential fragments.
4. Confirm `.gitignore` still excludes runtime/temp artifacts.

## Incident Procedure

If sensitive data was pushed:
1. Revoke/rotate compromised credentials immediately.
2. Remove exposed files from the default branch in a follow-up commit.
3. If required by organizational policy, rewrite history and force-push with approval.
4. Notify maintainers and affected stakeholders.

## Ownership

Repository maintainers are responsible for enforcing this policy during code review and release workflows.
